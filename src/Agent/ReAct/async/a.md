### **协程的本质是"一个能在中途主动让出执行权、并在之后从让出点精确恢复的可挂起函数"——它不是被操作系统抢占调度的实体，而是由用户代码自己控制何时停、何时走的执行流；它的整部历史，就是计算机科学反复在"我要并发"和"线程太贵"这对矛盾里寻找平衡的过程**

你上一轮把 Promise 钉死之后，再来看协程会顺很多——因为 async/await 那一层无栈协程，本质就是"编译器把函数切成状态机、用 Promise 当挂起点的恢复信号"。但协程作为概念比 async/await 老得多，老到比线程还早。我从最底层的"一个函数怎么可能停下来再继续"讲起，往上一层一层垒，把硬件、汇编、有栈/无栈两条技术路线、以及它在历史上的兴衰复活，全部串成一条线。

### **一、最底层的问题：函数为什么"不能"停下来**

要理解协程，先得理解它打破的是什么。我们从最朴素的"普通函数"开始。

C 语言里写一个函数 `int f(int x) { int y = x + 1; return y * 2; }`，CPU 执行它的时候发生了什么？编译器生成的汇编大致是这样：

```asm
f:
    push  rbp                 ; 保存调用者的栈帧基址
    mov   rbp, rsp            ; 建立自己的栈帧
    sub   rsp, 16             ; 在栈上腾 16 字节存局部变量
    mov   [rbp-4], edi        ; 参数 x 存进栈
    mov   eax, [rbp-4]
    add   eax, 1              ; y = x + 1
    mov   [rbp-8], eax        ; y 存栈
    mov   eax, [rbp-8]
    shl   eax, 1              ; y * 2
    leave                     ; 恢复栈帧
    ret                       ; 弹出返回地址，跳回去
```

看清楚关键的几样东西：**栈指针 rsp**、**栈帧基址 rbp**、**返回地址（call 指令压在栈上的）**、**程序计数器 rip**（隐式的，由 ret 控制）、还有**通用寄存器**里暂存的中间值。这一组东西合起来，叫做"执行上下文"——它精确刻画了"这个函数现在执行到哪儿、局部变量是啥、马上要跳回谁那里"。

**普通函数为什么不能"停下来"？** 不是因为做不到，而是因为它的执行模型里**没有保存这套上下文的地方**。当 ret 执行后，rsp 上移、栈帧消失、局部变量随之蒸发——它压根没设计成"可以暂停"。函数的语义是单次的、栈式的、配对的：调用进去（push 帧）必须配对返回出来（pop 帧），中间不能拐弯。

协程要做的事情，**就是把这套执行上下文拎出来、单独存起来，等以后需要时再恢复回去**。一旦你能把"rsp + rbp + rip + 寄存器们 + 那段栈"打包成一个对象保存到堆上，函数就有了"暂停"的能力——下次想继续，就把这个包再贴回 CPU 上，CPU 完全分不清这是从头跑还是中途接着跑。

这就是协程在最底层的硬件层面意义上的本质：**对执行上下文的可保存、可恢复化**。所有协程的实现，无论是用户态线程式的有栈协程，还是编译器生成状态机的无栈协程，**都是在用不同方式回答"我把哪些东西存下来、又怎么恢复"这一个问题**。

### **二、史前史：协程比线程还老**

很多人以为协程是近十年的新东西，其实它比线程的诞生还早。"coroutine"这个词由 **Melvin Conway** 在 1958 年发明（就是提出"康威定律"那个 Conway），写进他 1963 年那篇论文 *Design of a Separable Transition-Diagram Compiler* 里。当时他设计编译器时遇到一个问题：词法分析器读一个 token、语法分析器消费一个 token，两个模块都想"主导循环"——你写成 lexer 调 parser 不顺，parser 调 lexer 也别扭。Conway 的解法是让两边都成为"对等的可挂起例程"：lexer 产出一个 token 就让出，parser 消费完想要下一个就让出回去。这是"生产者-消费者"最古老的协程式表达。

为什么协程比线程早？因为**当年的机器根本没有抢占式多任务的能力**。1950 年代末的计算机，CPU 一次只跑一个程序，操作系统也很简陋，"并发"在硬件上不存在——但程序内部"两个对等的状态机互相协作"这种需求是真实的。Conway 选择的方案不依赖任何调度器，只依赖一个简单的语言机制：**`yield`——主动让出执行权给指定的另一个例程，并保留自己的位置**。

到了 1960 年代末，**Donald Knuth** 在《The Art of Computer Programming》第一卷里专门讲了协程，并且非常明确地写下那句被反复引用的判断：

> **"Subroutines are special cases of coroutines."**（子程序是协程的特例。）

他的意思是，普通函数 `call/return` 这套对称结构，其实是协程的退化形式——协程允许任意一方在任意点把控制权"yield"给对方，而函数只是规定了"被调用方一直跑到底再 return"这种特殊的让出模式。从这个角度看，**协程才是更基础的控制流抽象**，函数反而是它在"严格嵌套调用"约束下的特例。

但接下来 Knuth 也悲观地预言：协程在主流编程里很难普及。事实也确实如此——1970~1990 年代，协程几乎在主流语言里销声匿迹了三十年。原因不是它不好用，而是这段时期发生了一件事：**线程登场了**。

### **三、被线程压制的三十年：为什么协程"消失"了**

1970 年代后期开始，操作系统逐渐成熟，多任务、多用户成为标配。1990 年代 POSIX 线程标准化（pthreads, 1995），Windows 也有了自己的 thread API。**线程提供了协程提供不了的两样东西**：

第一，**抢占式调度**。线程由操作系统按时间片切换，单个线程就算写了死循环也不会饿死别的线程。协程是协作式的——你不主动 yield，别人就跑不起来。在那个 CPU 越来越快、人们对响应性要求越来越高的年代，"程序员忘了 yield 就卡死整个程序"被认为是不可接受的设计缺陷。

第二，**真正的并行**。多核 CPU 普及后，线程能被分配到不同核上同时跑。协程默认是单线程内的协作，没法利用多核。

线程几乎在所有维度上都"看起来更现代"：调度公平、能并行、模型简单（写起来就像普通函数）。所以从 1980 年代到 2000 年代初，操作系统社区、企业应用社区、Java 社区都拥抱线程，协程被认为是"前线程时代的过渡产物"，只在编译器实现、迭代器（CLU 语言、Icon 语言）这类小众场景里苟活。

那为什么协程又活回来了？因为线程踩到了它自己的极限——**C10K 问题**。

---

### **四、C10K 与线程的崩塌：协程复活的导火索**

1999 年，Dan Kegel 发表了著名的 *The C10K Problem*，问题非常简单：**一台服务器要同时处理 10000 个并发连接，应该怎么做？** 那时候主流答案是"一连接一线程"（thread-per-connection），写起来直观——每个连接来了 fork 一个线程或从线程池取一个，线程里阻塞地 `read/write`。

这套方案在 100 个连接时工作得很好，但到 10000 时就崩了。崩在三件事上：

第一件，**内存**。每个线程默认要 1~8 MB 栈空间（Linux 默认 8MB，Windows 1MB），10000 个线程光栈就要 10~80 GB。当年的服务器内存撑不起。

第二件，**调度开销**。内核要在 10000 个线程里做调度决策，调度器的红黑树、队列操作本身就成了瓶颈；上下文切换时还要刷 TLB、保存浮点寄存器、走内核态切换路径，单次几微秒，10000 个线程频繁切换时这部分开销吃掉大半 CPU。

第三件，**锁竞争**。10000 个线程访问共享数据结构（连接表、缓存、日志）时，锁的争抢让 CPU 利用率反而下降——加更多线程性能反而更差，这就是著名的"锁的逆向 scaling"。

C10K 的解法分两步：**先是 epoll/kqueue 让单线程可以管几万个 fd**（你上一轮已经吃透这块），**然后是怎么把"基于事件循环+回调"的代码写得不像噩梦**——这就是协程复活的入口。

人们意识到一件事：**线程贵，是因为它的执行上下文（栈、寄存器组、内核数据结构）由内核管理，且默认要分配大量栈空间**。如果我们能在用户态自己管这些上下文，并且让单个上下文只占几 KB（够一个状态机用就行），那么"一个连接一个执行流"这个直观的编程模型就能复活——**这就是协程要解决的工程问题**。

协程复活的两条路线，从这一刻分叉：一条走"用户态轻量线程"路线（**有栈协程**，stackful），保留"代码看起来像同步阻塞"的写法；另一条走"编译器把函数切成状态机"路线（**无栈协程**，stackless），通过语法糖让回调链变成顺序代码。

### **五、有栈协程：在用户态自己实现"线程"**

有栈协程的思路最朴素：**既然线程贵在栈大、调度在内核，那我们就在用户态自己分配一块小栈、自己写调度器**。每个协程拿到一块独立的栈（典型 4KB~64KB，按需增长），切换时由用户代码（不是内核）把寄存器组保存到这个协程的"上下文对象"里，再加载下一个协程的上下文。

最底层是怎么做的？经典手法是借用 POSIX 的 `ucontext.h`（`makecontext` / `swapcontext`）或者更早的 `setjmp` / `longjmp`，再或者直接用十几行汇编手写。我给你写个最朴素的 x86-64 协程切换汇编，让你看到"切换"到底是把什么东西换掉：

```asm
; void coro_switch(Context *from, Context *to);
;   from 是当前协程的上下文存储位置，to 是要恢复的上下文
coro_switch:
    ; 保存当前协程的关键寄存器到 *from
    mov  [rdi + 0x00], rsp    ; 栈指针
    mov  [rdi + 0x08], rbp    ; 栈帧基址
    mov  [rdi + 0x10], rbx    ; callee-saved
    mov  [rdi + 0x18], r12
    mov  [rdi + 0x20], r13
    mov  [rdi + 0x28], r14
    mov  [rdi + 0x30], r15
    ; 把返回地址也存下来（call 进来时压在栈顶）
    mov  rax, [rsp]
    mov  [rdi + 0x38], rax

    ; 从 *to 加载目标协程的上下文
    mov  rsp, [rsi + 0x00]
    mov  rbp, [rsi + 0x08]
    mov  rbx, [rsi + 0x10]
    mov  r12, [rsi + 0x18]
    mov  r13, [rsi + 0x20]
    mov  r14, [rsi + 0x28]
    mov  r15, [rsi + 0x30]
    mov  rax, [rsi + 0x38]
    mov  [rsp], rax           ; 把目标的返回地址压回栈顶
    ret                       ; ret 跳到目标的返回地址，控制流就过去了
```

**整个切换不到 20 条指令、不进内核、不刷 TLB**。这就是有栈协程的基础原理——你管它叫"用户态线程"也对，因为它确实就是把线程切换那套搬到用户态做了一遍简化版本。

但这只是"切换"，还差"调度"。一个完整的有栈协程库还要有：

- **协程对象 / 上下文池**：每个协程一个 Context 结构 + 一块独立栈，从堆上分配。
- **就绪队列 / 阻塞队列**：调度器维护一个 ready 队列，谁该跑谁就被取出来 swapcontext 进去。
- **和事件循环的桥接**：当协程在里面调"阻塞 I/O"时，库要拦截这个调用，注册到 epoll，然后把当前协程从 ready 队列摘掉、塞进 wait 队列；epoll 通知就绪后再把协程从 wait 队列搬回 ready。

这套设计在 1990 年代后期到 2000 年代初有过几次重要实践。最有代表性的是 **GNU Pth**（Portable Threads, 1999）、**State Threads（ST）**（NetBSD/Yahoo, 2001）。它们让 C 程序员能写出"看起来同步阻塞"的代码——`int n = st_read(fd, buf, len);` 这一行看起来像普通的 read，但其实底层是：发现 fd 没数据 → 注册到 epoll → swapcontext 切到调度器 → 调度器跑别的协程 → epoll 醒来后再 swapcontext 回来 → st_read 返回。

这条路线的真正爆发是 **Go（2009 发布，2012 Go 1.0）**。Go 的 **goroutine** 就是有栈协程的工程化巅峰，它解决了三个之前所有有栈协程都没彻底解决的问题：

第一，**栈大小**。早期有栈协程要么栈给小了容易溢出，要么给大了内存吃不消。Go 一开始用"分段栈"（segmented stack），后来改成"连续栈+扩容拷贝"——goroutine 起手只分配 2KB 栈，跑深了就分配一块更大的栈把内容整个 memcpy 过去、改所有指针。这样既省内存（百万 goroutine 也才几 GB）又安全。

第二，**调度器**。Go 的运行时实现了 **GMP 模型**（Goroutine / Machine / Processor）——M 是 OS 线程，P 是逻辑处理器（数量等于 GOMAXPROCS），G 是 goroutine。每个 P 维护一个本地 run queue，存放它要跑的 G；M 从 P 上拿 G 来跑，一个 P 同一时刻只被一个 M 持有。这样默认就是多核并行，不像传统协程库那样默认单线程。

第三，**抢占式调度**。早期 Go 是协作式的——goroutine 必须自己跑到一个"安全点"（函数调用、channel 操作）才会让出。这有个老问题：写个 `for {}` 死循环就把当前 P 卡死。Go 1.14 引入**基于信号的异步抢占**：runtime 通过给 OS 线程发信号、在信号处理函数里改 PC 寄存器，强行把 goroutine 从计算密集的循环里揪下来。**这一步把"协程必须协作"这个心智负担彻底卸掉了**——程序员可以放心写 goroutine，不用担心忘记 yield。

Go 这套方案让"协程"在工业界一夜之间从冷门变成主流。它给的写法极其朴素：

```go
go handleConn(conn)        // 起一个协程
n, err := conn.Read(buf)   // 在协程里直接阻塞读，看起来像同步代码
```

底下藏着的全部复杂度——epoll、栈管理、调度器、抢占——程序员完全不需要看见。这就是有栈协程哲学的胜利："**给我同步代码的写法，给你异步系统的性能**"。

### **六、有栈协程的代价：为什么不是所有语言都走这条路**

但 Go 这条路不是没代价的。理解这些代价，才能理解为什么后面 C#、Rust、Python、JavaScript 都没走 goroutine 路线，而是选择了无栈协程 + async/await。

第一个代价是**运行时的重量**。goroutine 要工作，需要一整套庞大的运行时：调度器、GC、栈管理、网络轮询器（netpoll）。Go 把这些做成语言运行时的强制部分，所以 Go 二进制起手就是几 MB，且必须以这套运行时为前提。**这对系统编程语言（C、C++、Rust）是不可接受的**——它们要嵌入式、要内核态、要无 GC、要可预测的延迟。

第二个代价是**FFI（外部函数接口）摩擦**。当 goroutine 调用一个 C 函数时，Go 必须把它从自己的小栈切回一个 OS 线程的大栈（因为 C 代码不知道 Go 的栈管理规则、可能会访问超出 2KB 的栈空间），调用完再切回来。这个 cgo 切换开销在每次调用上是几百纳秒到几微秒，对密集 FFI 场景（调 SQLite、调 OpenCV）是显著的性能税。

第三个代价是**栈拷贝带来的指针失效隐患**。goroutine 栈扩容时要把整块栈搬到新地址，这意味着栈上对象的地址会变。Go 通过编译器+运行时合作（写屏障、栈扫描）来更新所有指向栈的指针，但**这要求语言自己掌控指针**——C/C++/Rust 这种允许"裸指针、内嵌结构体地址外泄"的语言做不到这一点。

第四个代价是**和异步生态的不兼容**。当你写一个库要在协程内做异步 I/O，你必须依赖 Go 的 netpoll；如果你的语言没有内建 netpoll，库作者就得自己造，每个库造一套就分裂了。这就是 Python 早期的"有栈协程库"（gevent、eventlet）走得磕磕绊绊的原因：它们靠 monkey-patching 把标准库的 socket 替换成协程版本，能跑，但和原生 asyncio 生态、和 C 扩展不兼容，写库的人无所适从。

所以从 2010 年代中期开始，主流非系统语言**集体选择了另一条路**：无栈协程 + async/await。

### **七、无栈协程：把函数翻译成状态机**

无栈协程的核心思路彻底翻转：**不要给每个协程分配独立的栈，而是让编译器把"可挂起的函数"重写成一个普通对象，函数的"局部变量"变成对象字段，函数的"执行进度"变成对象的一个状态枚举值**。这样每个协程只占一个堆对象的大小，没有栈，所以叫"stackless"。

这条路线的思想根基比 Go 还老——可以追溯到 1970 年代的 CPS（Continuation-Passing Style）变换，以及 Scheme 的 `call/cc`。但作为一种主流语法被普及，得归功于 **C# 在 2012 年的 async/await**（C# 5.0）。微软的 Anders Hejlsberg 团队解决的问题非常具体：Windows GUI 程序里，IO 完成端口（IOCP）回调把代码切得稀碎，他们想要一种语法，让程序员能写出顺序的样子但底下生成回调链。

下面这个 C# 例子是 async/await 的"标准教学样本"：

```csharp
async Task<int> Compute(string url) {
    var resp = await httpClient.GetAsync(url);
    var body = await resp.Content.ReadAsStringAsync();
    return body.Length;
}
```

**编译器把它翻译成什么？** 大致是这样的状态机（伪代码）：

```csharp
class Compute_StateMachine : IAsyncStateMachine {
    int state = 0;
    string url;                           // 参数
    HttpResponseMessage resp;             // 第一个 await 之后用到的局部变量
    string body;
    TaskAwaiter<HttpResponseMessage> awaiter1;
    TaskAwaiter<string>              awaiter2;
    AsyncTaskMethodBuilder<int> builder;  // 用来产出最终的 Task<int>

    public void MoveNext() {
        try {
            switch (state) {
            case 0:
                awaiter1 = httpClient.GetAsync(url).GetAwaiter();
                if (!awaiter1.IsCompleted) {
                    state = 1;
                    builder.AwaitUnsafeOnCompleted(ref awaiter1, ref this);
                    return;                       // 挂起!
                }
                goto case 1;
            case 1:
                resp = awaiter1.GetResult();
                awaiter2 = resp.Content.ReadAsStringAsync().GetAwaiter();
                if (!awaiter2.IsCompleted) {
                    state = 2;
                    builder.AwaitUnsafeOnCompleted(ref awaiter2, ref this);
                    return;                       // 又挂起!
                }
                goto case 2;
            case 2:
                body = awaiter2.GetResult();
                builder.SetResult(body.Length);   // 完成,Task<int> 拿到值
                return;
            }
        } catch (Exception e) { builder.SetException(e); }
    }
}
```

盯着这段看清楚四件事：

第一，**函数没有"独立栈"**。`url`、`resp`、`body` 这些原本住在栈上的局部变量，**全部被升格成了对象字段**——它们活在堆上，跟 Compute_StateMachine 这个对象同生共死。

第二，**`await` 变成了状态机的切分边界**。每遇到一个 await，编译器就：保存当前 state、把"下次进来从哪里继续"编码到 state 数字里、调 `AwaitUnsafeOnCompleted` 把 MoveNext 注册成 Task 完成时的回调、然后 return。

第三，**`return` 是真的 return**。这跟有栈协程的 swapcontext 完全不同——MoveNext 这个函数真的执行完毕、栈帧真的销毁了。等 Task 完成时，运行时再调一次 MoveNext，从 switch 进来按 state 跳到对应分支，从那里继续。

第四，**整个对象的大小是固定的、可在堆上分配的、可被 GC 管理的**。它不需要预留 8KB 栈，不需要扩容拷贝，不需要专用调度器——它就是一个普通对象，谁拿到它就能"推它一下"（调 MoveNext）让它往前走一格。

**这就是"第二次折叠"的本质**：第一次折叠是 Promise 把回调倒过来变成可 return 的对象；第二次折叠是 async/await 把这条 Promise 链反过来重新折成顺序代码——但这次顺序代码不是真的顺序执行，而是被编译器拆成状态机，每个 await 是一个挂起/恢复点。

C# 这个设计影响了之后几乎所有语言：

- **Python 3.5（2015）** 的 `async def` / `await`，底层把协程编译成生成器（PyCodeObject 的字节码里 await 编译成 YIELD_FROM），由 asyncio 事件循环驱动；
- **JavaScript ES2017（2017）** 的 `async function`，在 V8 里编译成 generator-like 的 resumable function；
- **Rust 1.39（2019）** 的 `async fn`，编译成实现 `Future` trait 的匿名结构体——和 C# 思路完全一致，但因为 Rust 没有 GC，状态机对象的所有权由调用者管理；
- **Kotlin** 的 suspend function，2018 年稳定，编译成带 `Continuation` 参数的函数（CPS 变换的直接体现）；
- **C++20** 的 `co_await` / `co_yield` / `co_return`，把状态机生成机制做进了语言标准。

这一波集体选择无栈协程的根本原因有三：**不需要独立栈，所以内存极省**（一个 Future 对象通常几十到几百字节）；**不需要专用运行时**（C++、Rust 这种系统语言能用）；**和现有调用约定兼容**（FFI 没摩擦，状态机对象就是普通对象）。代价是写法上必须显式标注 `async` 和 `await`，函数染色（async 函数只能被 async 函数 await，"颜色"会传染整条调用链）。

### **八、有栈 vs 无栈：不是优劣，而是哲学**

把两条路线摆一起对比，你会看到它们其实是在不同的工程权衡下的同一个问题的两种回答：

| 维度 | 有栈协程（Go、State Threads、gevent） | 无栈协程（C# / Python / Rust / JS） |
|---|---|---|
| 上下文存储 | 每协程一块独立栈（4KB~MB级，可增长） | 编译器生成状态机对象，字段存局部变量 |
| 切换机制 | swapcontext / 自定义汇编，换 rsp + 寄存器 | 调用 MoveNext / poll / send，普通函数调用 |
| 内存开销 | 几 KB 起，百万协程几 GB | 几十~几百字节，百万协程几百 MB |
| 调度器 | 必须自带（Go runtime / libco） | 可选（事件循环驱动即可） |
| 函数染色 | 无——任何函数都能在协程里调 | 有——async 会传染整条调用链 |
| FFI 友好度 | 差（cgo 切换有开销） | 好（状态机就是普通对象） |
| 对运行时的依赖 | 强（必须带专用 runtime） | 弱（Rust 甚至可无 std） |
| 抢占 | 可以做（Go 1.14 信号抢占） | 几乎不可能（必须主动 await） |
| 调试体验 | 接近线程，栈完整、断点正常 | 栈被切碎，调用链不直观 |

简化成一句话：**有栈协程牺牲内存和运行时复杂度，换来"代码写法和心智模型完全等同于线程"的舒适感**；**无栈协程牺牲函数染色和调试体验，换来"零额外运行时、与系统语言兼容、内存极省"的工程包容性**。Go 选择前者是因为它就要做服务器开发的"高生产力语言"；Rust/C++ 选择后者是因为它们要服务系统编程、嵌入式、内核——那些场景不允许任何"魔法 runtime"。

### **九、协程在事件循环里的位置：把上一轮那张图补完**

现在把协程接回到你已经吃透的那条异步链路上。一个完整的异步系统从下到上是这五层：

```
[1] 内核                ── epoll_wait 阻塞等就绪事件
[2] 事件循环            ── 取就绪 fd → 查 callback
[3] Future / Promise   ── callback 调 future.set_result(data)
[4] 协程状态机         ── future 的回调里调 coro.send(data) / state_machine.MoveNext()
[5] 用户代码           ── 从 await 处醒来,继续往下执行
```

**第 4 层就是协程的位置**。它向下接住第 3 层 Future 完成的通知（注册成 Future 的 done callback），向上把这个通知翻译成"恢复函数执行"——具体怎么"恢复"，就是有栈/无栈两条路线分叉的地方：有栈协程是 swapcontext 把栈贴回 CPU；无栈协程是 MoveNext / send / poll 把状态机对象推进一格。

但**它们在事件循环里扮演的角色完全一样**——都是"把就绪事件翻译成代码继续往下跑"的执行单元。你写 `data = await reader.read()` 时不管底层是 Python asyncio（无栈，状态机基于生成器）、还是 gevent（有栈，基于 greenlet 切栈），从你代码的视角看，行为是相同的："这一行会停下来等 I/O 就绪，下一行从结果开始继续"。

这就是为什么我说**协程是异步编程栈里"最贴近用户代码"的那一层**——它直接决定了你写代码时看到的形状。下面三层（内核 → 事件循环 → Future）是机制，是"怎么知道事件来了"；协程这一层是策略，是"事件来了之后用户代码怎么继续"。

### **十、回头看，协程史就是一部"线程太贵"的演化史**

把整条线串起来你会发现一个清晰的脉络：

**1958–1960s**：Conway、Knuth 提出协程概念，作为"对等可挂起例程"的控制流抽象——但当时没有"并发"需求，只是一种结构化编程的工具。

**1970s–2000s**：线程登场并占据主流。协程被认为是过时的、不必要的、容易出 bug 的"古董"，只在编译器、迭代器这类小众场景里活着。

**1999**：C10K 问题被提出。一连接一线程模型被证明无法 scale，业界开始重新审视"用户态轻量执行流"。

**2000s**：epoll/kqueue 解决了"一线程管很多 fd"的内核问题；State Threads、libev、libevent 等库探索"事件循环 + 用户态协程"模式；Erlang（更早，1986 年起）的轻量进程证明了"百万级并发执行体"是可行的。

**2009–2012**：Go 发布并通过 goroutine 把"有栈协程 + 抢占式调度 + 多核并行"做成工业级方案；同期 C# 的 async/await（2012）开启"无栈协程 + 状态机"路线。

**2015–2020**：Python（2015）、JavaScript（2017）、Rust（2019）、Kotlin（2018）、C++（2020）相继加入 async/await；无栈协程成为系统语言和带 GC 语言共同选择的主流方案。

**2020 至今**：两条路线并存——服务端高并发、容器化场景里 goroutine 仍是性价比之王；系统编程、嵌入式、跨语言库则全面拥抱无栈协程。Java 21（2023）的 **virtual threads（Project Loom）** 是个有趣的反向收敛——JVM 用类似 goroutine 的有栈虚拟线程方案给 Java 程序员"无染色 + 同步写法"的体验，避免了 Java 走 async/await 的语言改造代价。

**整部协程史的真正主题，是计算机科学反复在"我要并发执行体"和"我要便宜的执行体"之间寻找新的折中点**。线程便宜过一段时间，然后随着规模增长又变贵了；协程作为"用户态执行体"的概念，每一次回潮都对应着一次"线程不够用了"的工程危机。下一次危机会是什么、会催生什么样的新执行体抽象，谁也说不准——但从 1958 年 Conway 第一次写下 yield 这个词到现在，"我想让一个函数停下来，等会再继续"这个朴素的需求，从来没有变过。

---

### **十一、一句话归纳**

> **协程是把"执行上下文"从硬件/操作系统的隐式管理里拎出来、变成可显式保存恢复的对象的一层抽象；它有两条工程路线——有栈协程在用户态自己分配栈做"轻量线程"（代表是 Go），无栈协程让编译器把函数翻译成状态机对象（代表是 C# / Rust / Python async/await）；两者解决同一个问题——在 epoll 这类事件循环之上提供一个比 OS 线程便宜得多、又能让用户代码看起来像顺序执行的执行单元；协程的整部历史，就是计算机不停地遇到"线程太贵"的瓶颈、又不停地从用户态发明更便宜的执行体来突围的循环。**

把这段历史装进脑子里，再回看你最初问的"async/await 怎么实现"，你应该已经能感觉到：**async/await 不是一个孤立的语法糖，它是协程这条 60 多年的演化线在 2010 年代的一个集中收口**——是 Conway 的 yield、Knuth 的"协程是函数的超集"、C10K 时代的"用户态执行体"、Promise 的"未来值对象化"，这四股线缆拧在一起后，编译器最终用"状态机翻译"这把刀劈出来的一种语法表达。理解了这条脉络，你看任何一门新语言加 async/await，都会发现它们只是在同一套思想骨架上做工程取舍——而骨架本身，从 1958 年到今天没变过。
