## 第一步：变量存在哪？

函数执行时，局部变量存在**栈（stack）**上：

```javascript
function foo() {
  let x = 10; // x 在栈上
  console.log(x);
} // 函数结束，栈帧销毁，x 没了
```

函数一结束，栈帧弹出，`x` 就**死了**，内存被回收。

## 第二步：函数里返回函数

```javascript
function outer() {
  let x = 10;

  function inner() {
    console.log(x); // inner 引用了 outer 的变量 x
  }

  return inner; // 把 inner 返回出去
}

const fn = outer(); // outer 执行完了，按理说 x 该死了
fn(); // 10 —— 但 x 还活着！
```

问题来了：`outer()` 已经执行完毕，栈帧应该销毁了，`x` 应该没了。但 `fn()` 调用时居然还能访问 `x`？

## 第三步：这就是闭包

引擎发现 `inner` 引用了 `outer` 的变量 `x`，于是做了一件事：

> **把 `x` 从栈上搬到堆上**，让它不随着 `outer` 的栈帧一起销毁。

`inner` 函数 + 它引用的外部变量 `x`，打包在一起，就叫**闭包（Closure）**。

```
闭包 = 函数本体 + 它捕获的外部变量
```

只要 `inner`（也就是 `fn`）还活着，`x` 就不会被垃圾回收。

## 第四步：闭包能记住状态

```javascript
function counter() {
  let count = 0;        // 被闭包捕获，搬到堆上

  return {
    add()  { count++; },
    get()  { return count; }
  };
}

const c = counter();    // counter() 执行完了，但 count 还活着
c.add();
c.add();
c.get(); // 2 —— count 一直在堆上活着，被两个方法共享
```

这里 `add` 和 `get` 两个函数都**闭包捕获了同一个 `count`**，所以它们共享同一份状态。

## 回到你的 generator 编译版本

```javascript
function myGeneratorCompiled() {
  let state = 0;  // ← 闭包捕获
  let a, b;       // ← 闭包捕获

  return {
    next: function() {  // ← 这个函数引用了 state、a、b
      switch (state) {
        case 0:
          a = 1;
          state = 1;
          return { value: a, done: false };
        // ...
      }
    }
  };
}

const gen = myGeneratorCompiled();
// myGeneratorCompiled() 执行完了，但 state、a、b 还活着
// 因为 gen.next 这个函数闭包捕获了它们
gen.next(); // state: 0 → 1
gen.next(); // state: 1 → 2，a 和 b 都还在
```

`state`、`a`、`b` 本来是 `myGeneratorCompiled` 的局部变量，函数执行完就该销毁。但因为返回的 `next` 函数引用了它们，引擎把它们**搬到堆上**保活。每次调 `gen.next()` 都在操作**同一份** `state`、`a`、`b`——这就是为什么状态机能跨多次调用记住走到了哪一步。

**一句话总结：闭包就是函数能记住并访问它被创建时所在作用域的变量，即使那个作用域早已执行完毕。**
