# Phase 4.1：滑动窗口短期记忆

## 1. 先看清问题

Phase 3 的 `AgenticRAG.query()` 每次都只发送 `system + 当前问题`。回答结束后，消息列表随即消失，因此它是一个单轮 Agent。

短期记忆的第一个任务，是在每次请求前把最近的对话放回上下文：

```text
system
→ 历史 user 1
→ 历史 assistant 1
→ 历史 user 2
→ 历史 assistant 2
→ 当前 user
```

## 2. 为什么不保留全部历史

全量历史会持续消耗 Token，最终超过模型上下文窗口。历史越长，无关信息也越容易干扰当前任务。

本步使用最简单的「按轮数滑动窗口」：

```text
最大 3 轮

[Q1, A1] [Q2, A2] [Q3, A3]
新增 [Q4, A4]
        ↓
[Q2, A2] [Q3, A3] [Q4, A4]
```

淘汰单位必须是完整的 `user + assistant` 问答，不能把一轮从中间切开。

## 3. 记什么，不记什么

本实现只保留：

- 用户最终提出的问题
- Agent 最终返回的答案

不保留 Agent 内部的 function calling、检索片段和中间工具结果。这些内部轨迹体积大，而且工具消息必须和对应的 assistant tool call 严格配对。跨轮保留它们既浪费 Token，又容易破坏消息结构。

## 4. 运行实验

```bash
uv run python src/AIAgent/RAG/phase4_main.py
```

依次尝试：

1. `我叫小林`
2. `我喜欢蓝色`
3. `我的名字是什么？`
4. 输入 `/memory` 查看三轮窗口
5. 再进行三轮无关对话，再次输入 `/memory`

注意：“模型最后是否答对”不是严格的窗口测试，因为模型可能猜测。`/memory` 显示的内容才是确定性证据。

## 5. 它的局限

- 按轮数不等于按 Token：一轮可能只有几个字，也可能有几千字。
- 被淘汰的重要信息会彻底消失。
- 进程退出后会话消失，不具备跨会话记忆。

这些正是后续 `Token-based Truncation`、`Summary Buffer` 和长期记忆要解决的问题。
