# RAG（检索增强生成）技术概述

## 1. 什么是 RAG

RAG (Retrieval-Augmented Generation) 是一种将信息检索与文本生成相结合的技术范式。它的核心思想是：在大语言模型生成回答之前，先从外部知识库中检索相关信息，然后将这些信息作为上下文提供给模型，从而生成更准确、更有据可依的回答。

RAG 最早由 Facebook AI Research (FAIR) 在 2020 年的论文《Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks》中提出。

## 2. RAG 解决的核心问题

### 2.1 幻觉问题 (Hallucination)

大语言模型有时会生成看似合理但实际上不正确的内容。这被称为"幻觉"。RAG 通过提供真实的参考资料，大幅减少了幻觉的发生。

### 2.2 知识时效性

模型的训练数据有截止日期。RAG 可以实时检索最新信息，使模型能够回答关于最新事件的问题。

### 2.3 私有知识

企业的内部文档、个人的私有知识不可能出现在模型的训练数据中。RAG 让模型能够访问和利用这些私有知识。

### 2.4 可溯源性

RAG 可以追溯答案的来源，用户可以验证回答的正确性。这对于需要高可靠性的场景（如医疗、法律）非常重要。

## 3. RAG 的核心流程

### 3.1 索引阶段 (Indexing)

1. **文档收集**：收集需要被检索的所有文档
2. **文档解析**：将各种格式（PDF、Word、HTML 等）转换为纯文本
3. **文本分块**：将长文档分割成较小的文本块（chunk）
4. **向量化**：使用 Embedding 模型将每个文本块转换为向量
5. **存储**：将向量和对应的文本存入向量数据库

### 3.2 检索阶段 (Retrieval)

1. **查询向量化**：将用户的查询转换为向量
2. **相似度搜索**：在向量数据库中搜索与查询最相似的文本块
3. **返回结果**：返回 Top-K 个最相关的文本块

### 3.3 生成阶段 (Generation)

1. **上下文构造**：将检索到的文本块组装成上下文
2. **Prompt 构造**：将上下文和用户问题组合成 Prompt
3. **LLM 生成**：调用 LLM 基于上下文生成回答

## 4. 文本分块策略

### 4.1 固定大小分块

最简单的方式。按固定字符数或 token 数切分文档，通常带有重叠（overlap）以避免在块边界丢失上下文。

- 优点：实现简单，块大小均匀
- 缺点：可能在语义中间切断

### 4.2 基于分隔符的分块

按自然的文本边界切分，如段落、章节标题等。LangChain 的 RecursiveCharacterTextSplitter 就是这种策略，它会依次尝试按段落、换行、句子、空格进行分割。

- 优点：较好地保持语义完整性
- 缺点：块大小可能不均匀

### 4.3 语义分块

使用 Embedding 模型计算相邻句子的语义相似度，在语义变化较大的地方进行切分。

- 优点：最好的语义完整性
- 缺点：计算成本高，实现复杂

## 5. Embedding 模型

### 5.1 什么是文本 Embedding

文本 Embedding 是将文本映射到高维向量空间的过程。好的 Embedding 模型能让语义相似的文本在向量空间中距离更近。

### 5.2 主流 Embedding 模型

- **OpenAI text-embedding-3-small/large**：闭源，通过 API 调用，效果好
- **BGE 系列 (BAAI)**：开源，中英文效果优秀，可本地部署
- **Jina Embeddings**：开源，支持超长文本（8192 tokens）
- **GTE (阿里)**：开源，多语言支持好
- **Cohere Embed**：闭源 API，专为检索优化

### 5.3 选择 Embedding 模型的考虑因素

1. **语言支持**：中文场景推荐 BGE 或 GTE
2. **维度与成本**：维度越高精度越好但存储和计算成本也越高
3. **最大 Token 长度**：决定了能处理的文本块大小上限
4. **部署方式**：API 调用更简单，本地部署更可控

## 6. 向量数据库

### 6.1 为什么需要向量数据库

传统数据库使用精确匹配或模式匹配（如 SQL LIKE）进行搜索。但语义搜索需要找到"意思相近"的内容，这需要在高维向量空间中进行近似最近邻搜索 (ANN)。

### 6.2 主流向量数据库

- **Chroma**：轻量级，适合原型和小规模应用
- **FAISS**：Meta 开源的高性能向量搜索库
- **Milvus/Zilliz**：分布式向量数据库，适合大规模生产环境
- **Pinecone**：全托管云服务
- **Weaviate**：支持多种搜索模式（向量、关键词、混合）
- **Qdrant**：Rust 编写，性能优秀

### 6.3 向量搜索的核心算法

- **暴力搜索**：遍历所有向量计算距离，精确但慢
- **IVF (Inverted File Index)**：先聚类，搜索时只查最近的几个聚类中心
- **HNSW (Hierarchical Navigable Small World)**：构建多层图结构，高效且精度高，是目前最流行的算法
- **PQ (Product Quantization)**：向量压缩，牺牲少量精度换取大幅减小存储和加速搜索

## 7. RAG 的评估

### 7.1 检索质量评估

- **Precision@K**：Top-K 结果中相关文档的比例
- **Recall@K**：所有相关文档中被检索到的比例
- **MRR (Mean Reciprocal Rank)**：第一个相关结果排名的倒数的均值
- **NDCG (Normalized Discounted Cumulative Gain)**：考虑排名位置的评估指标

### 7.2 生成质量评估

- **Faithfulness（忠实度）**：回答是否忠实于检索到的上下文
- **Answer Relevance（答案相关性）**：回答是否回答了用户的问题
- **Context Relevance（上下文相关性）**：检索到的上下文是否与问题相关

### 7.3 常用评估工具

- **RAGAS**：开源的 RAG 评估框架，自动计算上述指标
- **DeepEval**：类似单元测试的 LLM 评估框架
- **TruLens**：提供 RAG 三角评估（Groundedness、Answer Relevance、Context Relevance）
